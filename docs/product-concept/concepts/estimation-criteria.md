# 見積もり基準

このドキュメントは、WorkHub でタスクチケットの見積工数を出すときの基準を定義する。実際の見積もりは AI の skill で行うが、ここではその前提となる考え方を整理する。

実装上の入出力や保存する情報は、[見積もり基本設計](../../basic-design/estimation-design.md) にまとめる。

## 🎯 見積もり基準の目的

WorkHub では、見積工数を「タスクそのものの大きさ」として扱う。担当者ごとの実力差や稼働可能時間は、見積工数ではなく、潜在能力や予想戦闘力で扱う。

これにより、タスク側の大きさと人側の処理能力を分けて管理できる。

## 🧑‍💻 基準実力と見積工数の考え方

基準実力とは、タスクの見積工数を揃えるための共通の物差しである。特定個人の能力ではなく、WorkHub 上で見積もりを比較できるようにするための基準として扱う。

基準実力を持つ人は、対象領域の基本的な知識と実装経験を持ち、既存の設計やコードを読みながら、標準的な品質でタスクを完了できる人を指す。

見積工数は、基準となる人がそのタスクを実行した場合に必要になりそうな作業時間である。担当者ごとの実力差は見積工数には含めず、潜在能力や戦闘力の計算で扱う。

難易度や得意不得意を別係数として持つのではなく、必要な調査、実装、確認、レビュー対応などを含めて、基準となる人なら何時間で終わりそうかを見積もる。

成果物、グループ、マイルストーンなど、配下タスクを持つチケットの見積工数は、配下タスクチケットの見積工数を合計して扱う。

## 🔍 AI が見積もるときに見る情報

AI は、タスクチケットの説明、目的、背景、完了条件、関連チケット、過去の判断履歴、対象画面や対象ファイル、想定される確認方法をもとに初期見積もりを行う。

見積もりに必要な情報が不足している場合は、無理に確定せず、不足している前提や確認事項を明示する。

## 🛠️ 人間が補正する観点

AI の初期見積もりは、最終的な確定値ではない。人間は、実装上の制約、既存コードの状態、外部依存、レビュー方針、チケット分解の粒度を踏まえて、必要に応じて見積工数を調整する。

見積工数を調整した場合は、なぜ調整したのかをチケットの説明や変更履歴に残す。これにより、次回以降の見積もりや振り返りに使える判断材料を蓄積する。

## 📦 AI見積もりに求める出力

見積もり skill では、このドキュメントの基準に沿って、タスクチケットごとに見積工数を出す。

AI は少なくとも、以下を出力できるようにする。

| 出力 | 内容 |
| --- | --- |
| 見積工数 | 基準となる人が実行した場合に必要な時間 |
| 見積根拠 | どの情報をもとに見積もったか |
| 不足情報 | 見積もりの確度を下げている未確認事項 |
| 分解提案 | タスクが大きすぎる場合の分割案 |
