# WorkHub タスクマネジメント・プレイブック

## 🌐 はじめに

**WorkHub は、社内のデジタル化プロジェクトに関わる人間、AI、外部ツール、コンテキストを一元的に紐付け、人間とAIが同じ文脈を共有しながら協働できるHubを目指す。**

本ドキュメントでは以下を定義する：

## 1. ❓ Why WorkHub（なぜこのツールが必要か）

### 1.1 🧩 情報が分散し状況把握が困難になる

社内のデジタル化プロジェクトでは、開発タスク、調査、資料作成、会議準備、突発依頼、学習・検証タスクなど、多様なタスクが同時に発生する。それらのタスクは GitHub、Notion、Google Drive、Slack、Calendar など複数のツールに分散しやすく、どのタスクがどこにあるのか分からなくなったり、過去の判断や現在の状況が追跡できなくなりがちである。

特に、**デジタル化や効率化を目的とするタスクやプロジェクトの管理自体こそ、優先的にデジタル化・効率化する必要がある。**

### 1.2 📊 優先順位と実行可能性が見えなくなる

タスクが増えるにつれ、何を優先すべきか、どこまで受けられるか、誰が対応可能かといった全体像が見えなくなる。結果として、重要なタスクが後回しになったり、過剰なコミットメントによって計画が破綻しやすくなる。

### 1.3 🤝 認識合わせのコストが高くなる

タスクの目的、背景、判断理由、完了条件などの情報が整理されていないと、依頼人と作業者の認識合わせに多くの時間が必要になる。また、AI に作業を依頼する場合も、背景や判断履歴、参照資料を毎回説明し直す必要があり、人間とAIの間でも認識合わせのコストが発生する。

## 2. 🧠 Core Concepts（核となる考え方）

### 2.1 🎫 チケット制を取り入れてすべてを紐付ける

WorkHub は、仕事を [チケット制](./concepts/ticket-management-system.md) で扱う。

各チケットは、内容、ステータス、完了条件などの基本情報だけでなく、マイルストーン、成果物、GitHub Issue / PR、Notion ページ、Google Drive ファイル、Slack スレッド、AI セッションなどの関連情報とも紐付く。

### 2.2 📈 リソース情報で実行可能性を判断する

WorkHub は、各チケットの見積工数、投下工数、進捗、担当者、期限と、担当者の稼働可能時間を突き合わせて、成果物やマイルストーンの達成時期を予測することが可能。

### 2.3 ✅ チケットごとにレビューしやすくする

WorkHub では、タスクをチケット単位に細分化して管理するため、レビュー対象とレビュー観点を整理しやすくなる。また、このツール上でレビューができる。

### 2.4 🤖 人間とAIのコラボレーションを支える

WorkHub では、人間とAIが同じチケット情報を参照して協働する。AI はチケットに紐付いた目的、背景、関連資料、過去の判断履歴、完了条件などを理解して、タスク遂行の支援を行うことができる。

### 2.5 🗂️ 管理区分でさまざまな仕事を扱う

WorkHub では、[管理区分](./concepts/work-management-categories.md) を使って、性質の異なる仕事を同じチケット管理の枠組みで扱う。

| 管理区分 | 扱う仕事 | 補足 |
| --- | --- | --- |
| Project | ゴールや成果物に向けて計画的に進める仕事 | スプリント単位で計画・実行・振り返りを行う |
| Ad-hoc | 突発依頼や単発対応 | Project に属さない仕事を扱う |
| Explore | 学習・検証・実験のための仕事 | 定量化しにくい探究活動も見積もり・実績を追跡する |
| Reserved | カレンダー上の拘束時間 | タスク管理の対象ではなく、実行可能性の判断に使う |

### 2.6 🧮 見積もりで計画の現実性を判断する

WorkHub では、[見積もり方法](./concepts/estimation-method.md) を定義し、チケットの見積工数、担当者の作業可能量、投下工数を突き合わせて計画の現実性を判断する。見積もりと実績の差分を振り返ることで、次の計画やチケット分解の精度を高める。
